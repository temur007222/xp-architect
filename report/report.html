Test methodology

We ran a formative moderated usability test — the goal was to surface design problems early, not to measure final performance. The protocol follows the lecture-11 outline:

FormatModerated, in-person · think-aloud protocol
Participants5 users (Nielsen rule — 5 users surface ~85% of usability issues)
RecruitmentConvenience sample · classmates and friends · all 18–28 yrs · mobile-app literate
DeviceFunctional click-through prototype hosted on GitHub Pages, viewed on a 390 × 780 phone-sized viewport (laptop)
Roles1 facilitator (reads scenario, prompts) · 2 note-takers (timing + observations)
Per session≈15 min: 2 min briefing · 3 × scenario · 3 min post-test interview
MeasuredCompletion rate · time-on-task · errors · qualitative comments
Success ruleTask is 'complete' if the user reaches the success state within 3 attempts and no facilitator help

2 · Test Scenarios

Scenarios follow the format used in the lecture slides — each scenario sets a context, states a goal, lists a concrete task, defines success criteria, and notes the screens involved. They cover the three high-priority flows of XP Architect: onboarding, core gameplay loop, and post-game / progression.

SCENARIO 1    First-time onboarding
ContextA friend recommended XP Architect to a player who has never used it before. They open the app for the first time and want to understand the concept before committing to a real session.
GoalDiscover what the game is about, learn how to play, and reach the first interactive gameplay screen.
Task given to user"You just downloaded XP Architect because a friend told you it's interesting. Sign in, figure out what the game is about, and start your first game session. Tell me out loud what you're thinking as you go."
Success criteriaUser reaches the Place Events screen (W4) with a chosen archetype, having visited either the Home screen (W2) or the How-to-Play tutorial (W12).
Screens involvedW1 Login · W2 Home/About · W12 Tutorial · W3 Timeline Setup · W4 Place Events
Expected flowW1 → W2 → (W12 optional) → W3 → W4
SCENARIO 2    Design a complete game session
ContextA returning player has 5 minutes between classes and wants to design one full satisfaction curve for a Casual archetype player, then check the final score.
GoalComplete the core gameplay loop — pick archetype, place 8 events on the timeline, finalise, and review the session summary.
Task given to user"You're back in the app. Design a complete game session for a Casual player archetype. Place all 8 events, lock in the last one, and tell me your final satisfaction score."
Success criteriaUser reaches the Summary screen (W6) showing a satisfaction score, having placed at least one event on the timeline and confirmed a final round event.

What changed since the wireframe version

The original Task-10 wireframe prototype was a single static HTML click-through. After running the formative test, we re-built it as a functional React + TypeScript app (Vite + Tailwind + @dnd-kit + Recharts + Zustand) so that scenarios behave like the real product:

  • Real drag-and-drop on W4 using @dnd-kit (PointerSensor for mouse, TouchSensor for mobile) — every tile placement triggers a live recompute of the satisfaction curve.
  • Real scoring per §3.4 of the spec — RMSE between player and target curve → 0–100 score → Bronze / Silver / Gold rank tier. Covered by a vitest unit suite (11 tests).
  • Persistent sessions via localStorage (key xp-architect-v1) — profile, sessions, badges, and settings survive reloads.
  • All three top-priority usability fixes from this test are implemented in the live prototype (see §5).

5 · Top 3 Insights — Prioritisation

Method: Severity × Frequency matrix

We use the prioritisation method covered in the lecture: each observed issue is scored on two axes, then ranked by their product. This is the standard formative-test triage used in Nielsen Norman Group research and matches the lecture-11 framework.

Severity 1 — Cosmetic (annoyance, no functional impact)
2 — Major (slows task, requires workaround)
3 — Critical (blocks task, user gives up or fails)
Frequency 1 — Rare (1 user affected)
2 — Common (2–3 users affected)
3 — Universal (4–5 users affected)
PriorityPriority = Severity × Frequency → range 1 to 9. Tie-breaker: higher severity wins; second tie-breaker: affects the core gameplay loop.

Selected top 3 — implemented in the live prototype

#1 · B1 · Priority 9Scenario 2 · Severity Critical · 3 · Frequency Universal · 3
Target satisfaction curve is invisible

What we saw: The dashed target overlay on the satisfaction graph (W4) is too thin and visually similar to the user's own score line. All 5 users at some point asked 'what am I aiming for?' — P5 abandoned the entire scenario because of this. The game's core feedback loop fails when users can't see the goal.

Fix shipped: Target curve is now 3 px pink-acc dashed, with a labelled TARGET callout at the rightmost data point and a one-line caption above the graph: 'Match the dashed line for the best score.' See src/components/SatisfactionChart.tsx.

#2 · A2 · Priority 6Scenario 1 · Severity Major · 2 · Frequency Universal · 3
How to Play is skipped by 4 of 5 first-time users

What we saw: On W2, the 'Start Designing' CTA is larger, brighter, and placed above the 'How to Play' button. Only P3 (a UX designer) opened the tutorial. The others then hit W3 not knowing what an 'archetype' was, slowing their session and creating uncertainty that compounded in Scenario 2.

Fix shipped: On the very first launch only, the app surfaces a 3-step onboarding modal before W2 becomes interactive. The flag player.hasSeenOnboarding is persisted to localStorage so it never appears again. See src/components/OnboardingModal.tsx.

#3 · B3 · Priority 6Scenario 2 · Severity Major · 2 · Frequency Universal · 3
Orange 'Lock final event' button reads as an error

What we saw: On W5, the warning-orange Lock button was meant to signal irreversibility. Instead, 3 of 5 users hesitated, and P2 explicitly said 'wait, is this an error?'. The warning colour creates hesitation at the most important confirm action of the entire game.

Fix shipped: The Lock button is now primary teal (confirm semantics). The irreversibility warning lives in an orange banner above the chart, and the actual confirmation is a ConfirmModal shown after tap ('Lock the satisfaction curve? … Yes, lock it'). See src/screens/W5Final.tsx.

6 · Reflection

What the test confirmed

The core mechanic of XP Architect — placing events on a timeline and watching a curve respond — is fundamentally engaging. All 5 users understood the drag-place metaphor within 30 seconds of reaching W4. The information architecture and navigation system designed in Task 8 also held up: the menu drawer (W7) covered every screen the users needed to reach.

Where the design failed

Two patterns emerged. First, visual hierarchy of feedback: users don't see what they should be optimising for (insight B1). The satisfaction graph is the most important UI element of the game and its target overlay needs to be impossible to miss. Second, colour semantics in CTAs: using warning-orange for the most important confirm button (B3) sends the wrong signal — colour must match action, not consequence.

Methodology limitations

  • Convenience sample. All participants were 18–28 and mobile-app literate. Older users or first-time smartphone users may surface different issues, especially around the hamburger menu (C1).
  • Single session per user. Returning-user behaviour (e.g. menu discoverability after 10 sessions) is speculative based on first-encounter data.
  • Self-reported times. Stopwatch start/stop was facilitator-controlled and may have ±2 seconds of error per measurement. The qualitative observations are more reliable than the time figures alone.
  • Previously: click-through-only prototype. The original wireframe version faked the drag-and-drop. For the next round of testing we now have a fully functional prototype (live at the URL above) so the timeline interaction will be evaluated on real touch input.

What was shipped after this test

  • ✅ All three top-priority fixes (B1, A2, B3) implemented in the live React build.
  • ✅ Three additional priority-4 / priority-6 fixes folded in: A1 (login hierarchy), B2 (progress bar on W4), C2 (chevron + bordered settings rows), C3 (per-session history detail view).
  • 🟡 Deferred for v2: C1 bottom-tab navigation (low cost, but would change information architecture — to be validated in next test).
  • 🟡 Deferred for v2: A3 placeholder text on login (cosmetic).

Next steps

  • Re-test the same scenarios with 5 fresh participants on the live prototype to confirm the fixes work and don't introduce new issues — particularly around the new onboarding overlay (A2) and the confirm modal (B3).
  • Validate the W4 drag-and-drop on real mobile devices (the earlier test was on a laptop emulating a phone viewport — now that drag is functional we can run real touch testing).
  • Decide on the bottom-tab navigation question (C1) in the next iteration based on observed behaviour.
